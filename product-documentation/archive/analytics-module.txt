Analytics Module Implementation Plan
(Status: Implemented — backend & frontend wired, Feb 2026)
Context
Truleado currently has stub analytics data (random numbers) for pre-campaign creator snapshots, and basic social profile scraping via Apify (Instagram) and YouTube Data API v3. There is no deliverable-level post analytics — agencies cannot see how their campaign content is actually performing after publication.

This plan builds a complete analytics pipeline that fetches real metrics for each tracked deliverable URL, stores time-series snapshots, computes derived performance metrics, and exposes campaign-level dashboards via GraphQL.

Key decisions:

Apify stays for creator roster scraping; ScrapeCreators added for deliverable post analytics
YouTube: Public Data API v3 only (no OAuth)
Manual trigger only (no auto-scheduling)
New env var: SCRAPECREATORS_API_KEY
Implementation Order
Step 1: Database Migration
File: supabase/migrations/00030_deliverable_analytics.sql

Create 4 new tables:

analytics_fetch_jobs — Background job tracker (analogous to existing social_data_jobs)

id, campaign_id (FK), deliverable_id (FK, nullable for campaign-wide), agency_id (FK)
status ('pending'|'processing'|'completed'|'partial'|'failed')
total_urls, completed_urls, failed_urls, error_message
tokens_consumed, triggered_by (FK users), timestamps
deliverable_analytics_raw — Immutable raw API responses (append-only, no UPDATE/DELETE)

id, job_id (FK), tracking_url_id (FK deliverable_tracking_urls), deliverable_id, campaign_id, creator_id
platform ('instagram'|'youtube'|'tiktok'), content_url, raw_response (JSONB)
api_source ('scrapecreators'|'youtube_data_api'), fetch_status, credits_consumed, fetched_at
deliverable_metrics — Normalized time-series snapshots (immutable)

id, raw_id (FK to raw), tracking_url_id, deliverable_id, campaign_id, creator_id, platform, content_url
Common metrics: views, likes, comments, shares, saves, reach, impressions
platform_metrics (JSONB) — platform-specific extras
calculated_metrics (JSONB) — engagement_rate, save_rate, virality_index, etc.
creator_followers_at_fetch (for virality index denominator)
snapshot_at
campaign_analytics_aggregates — Upserted campaign rollups (UNIQUE on campaign_id)

Totals: total_views, total_likes, total_comments, total_shares, total_saves
Rates: weighted_engagement_rate, avg_engagement_rate, avg_save_rate, avg_virality_index
Cost: total_creator_cost, cpv, cpe
Breakdowns: platform_breakdown (JSONB), creator_breakdown (JSONB)
Deltas: views_delta, likes_delta, engagement_rate_delta
RLS: Agency-scoped via belongs_to_agency() for jobs, has_campaign_access() for metrics/raw/aggregates. No UPDATE/DELETE policies on immutable tables.

Indexes: On campaign_id, deliverable_id, tracking_url_id, snapshot_at DESC.

Reference patterns: supabase/migrations/00015_social_analytics.sql, supabase/migrations/00021_deliverable_tracking.sql

Step 2: Platform URL Detector
File: src/lib/analytics/platform-detector.ts

Pure functions, no dependencies:

detectPlatform(url) → 'instagram' | 'youtube' | 'tiktok' | null — hostname matching
parseTrackingUrl(url) → { platform, originalUrl, normalizedUrl, contentId }
extractInstagramShortcode(url) — regex /p/{code}/ and /reel/{code}/
extractYouTubeVideoId(url) — handles ?v=, youtu.be/, /shorts/
extractTikTokVideoUrl(url) — returns cleaned URL (SC takes full URL)
Step 3: ScrapeCreators API Client
File: src/lib/analytics/scrapecreators.ts

Follows pattern of src/lib/social/youtube.ts (generic GET helper + typed wrappers).

scGet(endpoint, params) — generic GET with x-api-key header, error handling
fetchInstagramPost(url) → GET /v1/instagram/post?url=<url>
Extracts: video_play_count, edge_media_preview_like.count (likes), edge_media_to_parent_comment.count (comments), taken_at_timestamp, owner.edge_followed_by.count
fetchTikTokVideo(url) → GET /v2/tiktok/video?url=<url>
Extracts from aweme_detail.statistics: play_count, digg_count, comment_count, share_count, collect_count, download_count
Error handling: typed RateLimitError for 429s, descriptive errors for other failures
Step 4: YouTube Video Fetcher
File: src/lib/analytics/youtube-video.ts

fetchYouTubeVideo(videoId) — calls videos?part=snippet,statistics,contentDetails&id={id}
Reuses ytGet pattern from src/lib/social/youtube.ts
Also fetches channel subscriber count for virality index
Returns { video: YouTubeVideoData, channelFollowers, raw }
Step 5: Response Normalizer
File: src/lib/analytics/normalizer.ts

Transforms raw platform responses into common shape:

normalizeInstagramPost(raw) — maps IG fields to common metrics
normalizeTikTokVideo(raw) — maps TikTok fields to common metrics
normalizeYouTubeVideo(raw) — maps YT fields to common metrics
normalizeRawResponse(platform, raw) — dispatch by platform
Output: { views, likes, comments, shares, saves, reach, impressions, platform_metrics, creator_followers_at_fetch }

Step 6: Metrics Calculator
File: src/lib/analytics/metrics.ts

Pure computation (no DB access):

computeDerivedMetrics(metrics) → { engagement_rate, save_rate, comment_rate, share_rate, virality_index, like_rate }
engagement_rate = (likes + comments + shares) / views
virality_index = views / creatorFollowers
Guards against zero denominators (returns null)
computeGrowthVelocity(current, previous) → deltas + percentage changes
Step 7: Campaign Aggregator
File: src/lib/analytics/aggregator.ts

aggregateCampaignMetrics(campaignId):
Query latest snapshot per tracking_url_id (DISTINCT ON)
Sum totals across all URLs
Compute weighted engagement rate, platform/creator breakdowns
Join campaign_creators for cost metrics (CPV, CPE)
Compare with existing aggregate for growth deltas
upsertCampaignAggregate(campaignId, result) — upsert into campaign_analytics_aggregates
Step 8: Barrel Export
File: src/lib/analytics/index.ts

Re-exports all modules from steps 2-7.

Step 9: Background Processing Route
File: src/app/api/analytics-fetch/route.ts

Follows pattern of src/app/api/social-fetch/route.ts:

POST handler, runtime = 'nodejs', maxDuration = 120
Auth via x-internal-secret header
Flow per job:
Load job + tracking URLs
For each URL (sequential, 500ms delay between):
detectPlatform(url)
Fetch via ScrapeCreators or YouTube API
Store raw in deliverable_analytics_raw
Normalize → compute derived metrics → store in deliverable_metrics
Update job progress
On error: store error, increment failed_urls, continue
Run aggregateCampaignMetrics after all URLs
Mark job completed/partial/failed
Exponential backoff for 429s (up to 3 retries, max 30s wait)

Step 11: Mutation Resolvers
File: src/graphql/resolvers/mutations/deliverable-analytics.ts

Token-gated flow (pattern from src/graphql/resolvers/mutations/analytics.ts):

fetchDeliverableAnalytics(deliverableId):

requireAuth + requireCampaignAccess(Permission.FETCH_ANALYTICS)
Load deliverable's tracking record + URLs
Token check: agency.token_balance >= urlCount
Deduct tokens, create analytics_fetch_jobs record
Fire-and-forget POST to /api/analytics-fetch
Log activity, return job
refreshCampaignAnalytics(campaignId):

Same auth + token flow
Loads ALL tracking records/URLs across campaign
Creates single campaign-wide job (deliverable_id = NULL)
Register in: src/graphql/resolvers/mutations/index.ts (add import + object entries)

Step 12: Query Resolvers
Add to: src/graphql/resolvers/queries.ts

campaignAnalyticsDashboard: Load aggregate row + per-deliverable breakdown + latest job
analyticsFetchJob / analyticsFetchJobs: Simple DB lookups (pattern from socialDataJob)
All queries use Permission.VIEW_ANALYTICS.

Step 13: Type Resolvers
File: src/graphql/resolvers/types.ts

Add field resolvers for AnalyticsFetchJob and DeliverableMetricsSnapshot (snake_case → camelCase mapping).

Step 14: Database Types
File: src/lib/supabase/database.types.ts

Add TypeScript types for all 4 new tables (Row, Insert, Update shapes).
